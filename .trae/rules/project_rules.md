# 🧩 高级前端设计师开发规则（Rule Set）

## 0. 核心原则（写在最前面，刻在脑子里）

代码不是"能跑就行"，而是**可读、可复用、可维护、可预期**

> 如果三天后你不想看自己的代码，那说明你今天写错了

---

## 1. 复用优先，禁止复制粘贴式开发

* 相同逻辑出现 **2次 → 抽函数**
* 相同结构出现 **2次 → 抽组件**
* 相同样式出现 **2次 → 抽变量 / class / token**

❌ 禁止：

```js
// 到处复制一模一样的逻辑
if (status === 1) { ... }
if (status === 1) { ... }
```

✅ 应该：

```js
const isActive = (status) => status === 1
```

---

## 2. 代码必须"自解释"，而不是靠猜

* 变量命名必须表达意图，不允许 `data` / `list` / `item` 泛滥
* 函数名必须是**动词 + 目标**

❌ 禁止：

```js
const fn = (a, b) => a + b
```

✅ 应该：

```js
const calculateTotalPrice = (price, quantity) => price * quantity
```

---

## 3. 注释规则（不是越多越好，是越准越狠）

只在以下情况写注释：

### ✔ 必须写注释：

* 复杂业务逻辑
* 反直觉处理
* hack / workaround
* 边界条件说明

### ❌ 禁止写：

```js
// 给变量赋值
const count = 10
```

### ✅ 正确示例：

```js
// 这里必须延迟300ms，否则动画与接口返回冲突导致闪烁
setTimeout(() => {
  updateUI()
}, 300)
```

---

## 4. 不允许"猜测式编程"

你不是占卜师，不要写"可能会用到"的代码

❌ 禁止：

```js
if (data?.user?.profile?.name) { ... } // 盲目防御
```

❌ 禁止：

```js
// 先写着 以后可能用
const tempCache = {}
```

✅ 应该：

* 明确数据结构来源
* 明确接口返回格式
* 不确定就打印 / 校验，而不是乱兜底

---

## 5. 组件设计原则（像搭积木一样优雅）

* 单一职责：一个组件只做一件事
* 可组合：通过 props 控制行为
* 不写"巨石组件"（超过300行就要警惕）

❌ 禁止：

```js
<SuperComponent /> // 里面干了10件事
```

✅ 应该拆成：

```js
<UserInfo />
<CourseList />
<ScheduleCard />
```

---

## 6. 样式规范（让UI有"秩序感"）

* 统一设计变量（颜色 / 圆角 / 间距）
* 禁止魔法数字

❌ 禁止：

```css
margin: 17px;
color: #123456;
```

✅ 应该：

```css
margin: var(--spacing-md);
color: var(--primary-color);
```

---

## 7. 状态管理原则

* 本地状态 → 组件内
* 跨组件 → 提升 or 全局
* 禁止"到处塞全局变量"

---

## 8. 错误处理（别装没看见）

* 所有 async 必须有 error 处理
* 用户必须知道发生了什么

```js
try {
  await fetchData()
} catch (err) {
  showError('加载失败，请稍后再试')
}
```

---

## 9. 性能意识（不是等卡了才优化）

* 避免重复渲染
* 列表必须有 key
* 大数据量考虑虚拟列表

---

## 10. Git & 提交规范（代码也有"记忆"）

* 一次提交只做一件事
* commit 信息必须清晰

✅ 示例：

```
feat: 添加课表卡片生成逻辑
fix: 修复课程时间显示错误
refactor: 拆分课程组件结构
```

---

## 11. 本地运行规则（这一条很硬，不准破）

⚠️ **项目永远只运行在 3000 端口**

* 不允许创建新端口
* 不允许 3001 / 5173 / 8080
* 如果冲突：

👉 **直接重启 3000**
👉 **kill 占用进程**
👉 **重新启动**

```bash
# Windows
netstat -ano | findstr :3000
taskkill /PID <pid> /F

# macOS/Linux
lsof -i :3000
kill -9 <pid>

npm run dev
```

> 3000 就像你的小窝，不换门牌号

---

## 12. 文件结构原则（别让项目变迷宫）

* 按功能拆分，而不是按类型
* 目录即语义

✅ 示例：

```
/course
  ├── components
  ├── hooks
  ├── services
  ├── utils
```

---

## 13. 最后一条（也是最重要的）

写代码的时候问自己一句：

> "这个写法，三个月后的我，会不会骂现在的我？"

如果答案是"会"

那就再改一版 🌙
